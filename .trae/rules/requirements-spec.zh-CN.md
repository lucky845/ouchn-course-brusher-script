# 开发需求规范 v1.1

## 核心规则

### [规则 1] 生成完整可运行代码（CRITICAL）
- 代码必须可立即执行，不允许占位符或 TODO
- 包含所有必要的 import、依赖和配置

### [规则 2] 复用现有代码和 API（HIGH）
- 创建新接口前检查现有代码库
- 不要重复发明已有功能

### [规则 3] 最小化新增依赖（HIGH）
- 优先使用项目现有依赖
- 添加新依赖需要充分理由

### [规则 4] 从之前的错误中学习（HIGH）
- 避免重复对话历史中的错误
- 一致应用修正

### [规则 5] 仅修改请求的内容（HIGH）
- 未经许可不要重构可工作的代码
- 最小化修改范围

### [规则 6] 验证所有 API 是否存在（CRITICAL）
- 使用前验证 API 是否存在
- 只使用已确认可用的 API

### [规则 7] 第一次就完全修复错误（HIGH）
- 解决根本原因，而不是症状
- 确保代码交付前能编译和运行

### [规则 8] 保持注释与代码一致（MEDIUM）
- 代码变更时更新注释
- 注释应反映实际行为

### [规则 9] 功能优先于完美（MEDIUM）
- 可工作的代码 > 完美的代码
- 先交付 MVP 再做增强

### [规则 10] 确保代码成功编译（CRITICAL）
- 交付前检查语法
- 解决所有编译错误

### [规则 11] 严格遵循提供的示例（MEDIUM）
- 匹配示例的代码风格和结构

### [规则 12] 尊重项目命名约定（MEDIUM）
- 遵循项目的命名风格

### [规则 13] 只使用真实存在的库（CRITICAL）
- 导入前验证包是否存在
- 确认库版本兼容性

## 关键规则摘要
1. 生成完整可运行代码（无 TODO、无占位符）
2. 复用现有接口/API
3. 最小化依赖
4. 只修改请求的内容
5. 使用前验证所有 API 是否存在
6. 代码必须能立即编译和运行
7. 只使用真实存在的库
